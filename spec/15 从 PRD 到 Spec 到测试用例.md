## 1. 为什么一定要把整条链路串起来看

前面你已经分别学过：

- `PRD`
- `User Story`
- `Spec`
- `Acceptance Criteria`
- 测试相关内容

如果只分别理解这些概念，你已经比很多新人清楚得多了。

但真实项目里，真正决定交付质量的，不只是“会不会写某一种文档”，而是：

- 这些文档之间能不能正确传递
- 同一条需求能不能从上到下一路收敛而不失真

因为现实里最常见的问题不是：

- 一种文档完全没写

而是：

- `PRD` 和 `Spec` 对不上
- `Spec` 和验收条件对不上
- 测试用例和需求目标对不上
- 最后做出来的功能和最初问题也对不上

所以这一章最核心的目标，是帮你建立一个完整链路视角：

`需求不是一份文档，需求是一条从问题定义到测试验证的收敛链。`

## 2. 为什么很多团队文档很多，交付还是失真

这是非常现实的问题。

有些团队表面上文档不少：

- 有 `PRD`
- 有原型
- 有规格
- 有测试用例

但最后仍然会出现：

- 产品觉得做偏了
- 开发觉得自己按文档做了
- 测试觉得很多情况没有定义

为什么？

因为这些文档虽然都存在，但它们之间没有稳定地对齐。

### 2.1 上层目标没有传到下层定义

例如：

- `PRD` 说的是提升运营处理效率
- 但功能 `Spec` 写成了纯页面调整

### 2.2 中间层规则没有传到测试

例如：

- `Spec` 写了状态限制
- 但测试用例没有覆盖这些状态

### 2.3 验收层没有和目标闭环

例如：

- 验收标准只检查“能不能点按钮”
- 却没有检查是否真的满足用户场景

这说明问题不在于“有没有文档”，而在于“文档链路是不是通的”。

## 3. 一条需求从想法到交付，中间到底经过哪些层

一个实用的理解方式是把它拆成下面几层：

1. 业务问题
2. 需求定义
3. `PRD`
4. `User Story`
5. 功能 `Spec`
6. `Acceptance Criteria`
7. 测试用例
8. 开发实现与测试执行

注意，这不是说每个团队都一定用这些完全一样的文档名，而是说思维上通常都会经过这些层次。

### 3.1 业务问题层

最开始通常是：

- 谁遇到了什么问题
- 为什么值得解决

### 3.2 需求定义层

开始回答：

- 这次要解决什么
- 目标是什么
- 边界是什么

### 3.3 文档收敛层

把上面的内容逐步写成：

- `PRD`
- `User Story`
- `Spec`
- 验收标准

### 3.4 验证落地层

最后再进入：

- 测试用例
- 实现
- 验证

如果你把这些层混在一起，后面就很容易失真。

## 4. PRD 在这条链路里负责什么

`PRD` 在整条链路里更像是“总定义入口”。

它主要负责：

- 解释为什么做
- 定义解决什么问题
- 说明目标和非目标
- 明确范围
- 列出主要功能方向

### 4.1 PRD 的价值

它能帮助团队建立对需求整体的共同理解。

### 4.2 PRD 的边界

它不一定足够细到直接支持每个复杂功能的开发。

尤其是当功能涉及：

- 多状态
- 多角色
- 多规则
- 多异常

通常还需要进一步写 `Spec`。

### 4.3 在链路里最容易出的问题

- `PRD` 只写目标，不写边界
- `PRD` 写了目标，但后续文档没有沿着目标收敛

## 5. User Story 在这条链路里负责什么

`User Story` 在这条链路里最重要的作用，是把需求持续锚定在用户价值上。

它负责回答：

- 谁想完成什么任务
- 从而获得什么价值

### 5.1 为什么这层不能省

因为团队很容易在进入实现细节时忘记：

- 这能力到底是为了谁
- 为什么要做

### 5.2 它的局限

`User Story` 不负责完整规则、异常、边界和验收。

它的作用是给方向，不是给全部定义。

### 5.3 在链路里最容易出的问题

- Story 写得好，但后续规格没有真正支撑它的价值

例如：

- Story 说要“快速定位待审核订单”
- 结果后续方案仍然操作复杂、查询很慢

这就说明链路断了。

## 6. 功能 Spec 在这条链路里负责什么

功能 `Spec` 是整条链路里最关键的收敛层之一。

它负责把上面的目标、价值和方向，继续写成：

- 行为定义
- 业务规则
- 状态变化
- 异常处理
- 边界条件

### 6.1 它像什么

它像一份“把功能从想法变成工程定义”的中间层文档。

### 6.2 它最重要的作用

是让开发和测试都不需要大面积靠猜。

### 6.3 在链路里最容易出的问题

- `PRD` 很完整，但 `Spec` 只剩页面描述
- `Story` 很清楚，但 `Spec` 没有把规则补齐

## 7. Acceptance Criteria 在这条链路里负责什么

这层负责把“完成”写成客观条件。

### 7.1 它的职责

- 说明做到什么程度算通过
- 把关键结果写成可验证表达

### 7.2 它和 Spec 的关系

- `Spec` 讲系统如何定义
- `Acceptance Criteria` 讲怎样判断这个定义已经满足

### 7.3 在链路里最容易出的问题

- `Spec` 很长，但验收标准很空
- 验收只覆盖主流程，不覆盖关键异常

## 8. 测试用例在这条链路里负责什么

测试用例不是再定义需求，而是把前面的需求和规格转化成可执行验证动作。

### 8.1 它负责什么

- 写清前置条件
- 写清测试步骤
- 写清输入数据
- 写清预期结果

### 8.2 它和验收标准的关系

- 验收标准定义“验证目标”
- 测试用例定义“验证步骤”

### 8.3 在链路里最容易出的问题

- 测试只按页面点点点，没有覆盖规则和边界
- 测试用例和规格脱节

## 9. 这些文档之间到底是怎么传递的

这一节非常关键。

你可以把它理解成一层层收敛。

### 9.1 从 PRD 到 User Story

`PRD` 里定义了需求方向后，可以提炼出一个或多个 `User Story`：

- 谁需要什么能力
- 为什么需要

### 9.2 从 User Story 到 Spec

`User Story` 再继续下沉，变成：

- 这个能力具体怎么工作
- 有哪些规则
- 有哪些边界

### 9.3 从 Spec 到 Acceptance Criteria

把规格里的关键行为和关键结果进一步提炼成：

- 哪些条件必须被验证通过

### 9.4 从 Acceptance Criteria 到测试用例

测试再根据验收条件和规格，把它展开成：

- 谁来测
- 怎么测
- 测哪些输入
- 期待什么结果

### 9.5 这条传递链为什么重要

因为每一层都不应该凭空重造，而应该从上一层继承并收敛。

如果每层都在“重新理解”，那失真几乎一定发生。

## 10. 一个完整例子：订单状态筛选从 PRD 到测试用例

这一节把完整链路串起来。

### 10.1 业务问题

运营人员在处理异常订单时，当前只能按订单编号搜索，无法快速定位待审核订单，导致处理效率低。

### 10.2 PRD 写法

背景：

- 当前订单后台缺少状态筛选能力，运营需要手工翻页查找待审核订单。

目标：

- 支持按订单状态筛选订单
- 缩短定位待审核订单的时间

范围：

- 本期支持单状态筛选
- 本期不支持多状态组合筛选

### 10.3 User Story

- 作为运营人员，我想按订单状态筛选订单，从而快速定位待审核订单。

### 10.4 功能 Spec

- 具有订单查看权限的运营和客服可使用状态筛选功能
- 筛选仅支持单选
- 用户选择状态并点击搜索后，系统仅返回符合状态的订单
- 无权限用户不得查看订单数据
- 查询结果为空时，页面展示空状态提示

### 10.5 Acceptance Criteria

- Given 用户已登录且具有订单查看权限
- When 用户选择“待审核”状态并点击搜索
- Then 系统应仅返回待审核订单

- Given 用户无订单查看权限
- When 用户访问订单列表页
- Then 系统应提示无权限
- And 系统不展示任何订单数据

### 10.6 测试用例

测试用例 1：

- 前置条件：使用具有订单查看权限的运营账号登录
- 步骤：
  1. 进入订单列表页
  2. 选择状态“待审核”
  3. 点击搜索
- 预期结果：
  1. 页面仅展示待审核订单

测试用例 2：

- 前置条件：使用无订单查看权限账号登录
- 步骤：
  1. 进入订单列表页
- 预期结果：
  1. 页面提示无权限
  2. 不展示任何订单数据

### 10.7 你应该看到什么

这一整条链路里，每一层都在回答不同问题，但又围绕同一个需求逐步收敛。

## 11. 第二个完整例子：退款申请从 PRD 到测试用例

### 11.1 业务问题

用户发现下错单后，当前只能联系客服人工申请退款，处理链路长、等待时间久。

### 11.2 PRD 写法

目标：

- 支持用户在线提交退款申请
- 降低客服人工处理压力

范围：

- 本期支持在线提交退款申请
- 本期不支持复杂分单退款场景

### 11.3 User Story

- 作为下单用户，我想在线提交退款申请，从而减少联系客服等待时间。

### 11.4 功能 Spec

- 仅已支付订单允许发起退款申请
- 用户本人可发起退款申请
- 单次退款金额不得高于实付金额
- 退款处理中订单不得重复提交退款申请
- 提交后退款状态更新为“申请中”

### 11.5 Acceptance Criteria

- Given 用户为订单本人且订单状态为已支付
- When 用户提交退款申请
- Then 系统应创建退款申请并将退款状态更新为“申请中”

- Given 订单退款状态为“处理中”
- When 用户再次提交退款申请
- Then 系统应阻止重复提交并提示当前状态不允许操作

### 11.6 测试用例

测试用例 1：

- 前置条件：用户登录，订单状态为已支付，退款状态为“无”
- 步骤：
  1. 进入订单详情页
  2. 点击申请退款
  3. 输入合法退款金额并提交
- 预期结果：
  1. 创建退款申请成功
  2. 退款状态更新为“申请中”

测试用例 2：

- 前置条件：订单退款状态为“处理中”
- 步骤：
  1. 再次提交退款申请
- 预期结果：
  1. 系统阻止提交
  2. 提示当前状态不允许操作

## 12. 为什么链路中任何一层写坏，后面都会跟着变形

这是这一章非常重要的结论。

### 12.1 PRD 写坏

如果问题定义就错了，后面所有文档都会在错误方向上细化。

### 12.2 User Story 写坏

如果用户价值表达偏了，后续方案可能虽然“做对了功能”，却没有解决正确问题。

### 12.3 Spec 写坏

如果规格缺规则、缺边界、缺状态，开发和测试都会靠猜。

### 12.4 Acceptance Criteria 写坏

如果完成定义不清楚，项目最后一定争议不断。

### 12.5 测试用例写坏

如果测试只覆盖主流程，关键规则再好也可能没有真正被验证。

所以链路不是独立模块，而是一环扣一环。

## 13. 常见断裂点

这一节可以当成高频风险清单。

### 13.1 目标没有传到规格

例如：

- 目标是“提高效率”
- 规格却只描述页面元素，没有定义能否真正缩短操作路径

### 13.2 规格没有传到验收

例如：

- `Spec` 写了无权限场景
- 验收标准却没有覆盖

### 13.3 验收没有传到测试

例如：

- 验收要求了空状态展示
- 测试用例没有设计空结果场景

### 13.4 测试验证结果没有回看需求目标

例如：

- 测试通过了
- 但上线后用户仍然说不好用

这说明测试验证了局部行为，却没有回看最初目标是否达成。

## 14. 如何判断一条需求的文档链路是不是闭环了

你可以用下面几组问题检查。

### 14.1 从上往下看

- `PRD` 定义的问题，是否在 `Story` 里体现了用户价值
- `Story` 里的能力，是否在 `Spec` 里写成明确行为
- `Spec` 的关键行为，是否在验收标准里被明确验证
- 验收标准，是否在测试用例里变成了具体步骤

### 14.2 从下往上看

- 测试用例每个关键验证点，是否都能回溯到验收标准
- 验收标准，是否都能回溯到规格定义
- 规格定义，是否都服务于最初需求目标

### 14.3 如果双向都能通

那这条链路就比较接近闭环。

## 15. 一个可直接复用的链路检查清单

下面给一个实用清单。

```md
## 需求链路检查清单

### 1. PRD
- 是否定义了问题、目标、范围

### 2. User Story
- 是否明确了角色、任务、价值

### 3. Spec
- 是否定义了行为、规则、状态、异常、边界

### 4. Acceptance Criteria
- 是否把关键结果写成可验证条件

### 5. Test Case
- 是否把关键验收条件转成具体测试步骤

### 6. 链路检查
- 上一层内容是否被下一层正确承接
- 下一层验证是否能回溯到上一层定义
```

这份清单的重点不在于形式，而在于帮助你看清：

- 文档是链路，不是孤岛

## 16. 这一章你应该带走什么

读完这一章，你至少应该记住下面几点。

第一，需求真正落地不是靠某一份文档单打独斗，而是靠整条文档链路协同。

第二，这条链路里各层大致分别负责：

- `PRD`：定义问题、目标、范围
- `User Story`：定义用户价值
- `Spec`：定义行为、规则、边界
- `Acceptance Criteria`：定义完成标准
- 测试用例：定义验证步骤

第三，很多团队文档不少但交付失真，根本原因不是“缺文件”，而是“链路断了”。

第四，判断一条需求是否真正闭环，不只是看它有没有文档，而是看这些文档之间能不能上下对齐、双向回溯。

## 小结

这一章最核心的判断是：

`写规格，不是在写一堆彼此独立的文件，而是在搭一条从业务问题一直通到测试验证的收敛链。`

只有这条链真正打通，文档才不只是“存在”，而是真的在驱动交付。
