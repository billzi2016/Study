# 本书 PRD：软件规格编写教程

## 1. 文档信息

- 文档名称：软件规格编写教程 PRD
- 文档类型：教程项目产品需求文档
- 当前状态：初稿
- 文档目的：定义这本教程为什么写、写给谁、写什么、怎么组织、做到什么程度算完成

## 2. 项目背景

这本书要解决的，不是“团队里已经很成熟的高级工程师如何进一步精炼规格表达”，而是一个更基础但更普遍的问题：

很多刚进入软件开发行业的人，知道“需求”“PRD”“Spec”“User Story”“验收标准”这些词，却并不真正理解它们之间的关系，也不知道怎样把一个模糊想法写成一份可以驱动设计、开发、测试和验收的规格。

现实里的常见情况是：

- 新人会写很多字，但写不出可执行的需求
- 文档看起来像说明文、会议纪要或页面描述，而不是工程规格
- 需求里只有主流程，没有异常、边界、状态和约束
- 产品、开发、测试对同一个功能理解不一致
- 文档无法直接支撑任务拆分、测试设计和验收

这本书的目标，就是系统性填平这一层认知断层。

它不是一本只讲术语定义的概念书，也不是一本只给模板的速查手册，而是一套面向新人、由浅入深、强调工程落地的规格写作教程。

## 3. 问题定义

当前目标读者在学习“怎么写 spec”时，通常会遇到以下问题：

### 3.1 不知道规格的本质是什么

很多人把规格理解为“把事情写清楚”，但没有意识到规格真正重要的是：

- 可执行
- 可验证
- 可讨论
- 可约束实现

也就是说，规格不是写得详细就行，而是要能进入工程流程。

### 3.2 不知道不同文档分别解决什么问题

新人常常混淆：

- PRD
- Spec
- User Story
- Use Case
- Acceptance Criteria
- Test Case

他们会把用户故事当成功能规格，把页面描述当成需求定义，把验收条件写成模糊口号。

### 3.3 不知道怎样从模糊想法逐步收敛

很多人会说：

- 支持退款
- 优化搜索
- 增强体验

但不知道继续追问什么，也不知道怎样把一句模糊描述，逐步收敛成：

- 角色
- 触发条件
- 输入输出
- 状态约束
- 业务规则
- 异常分支
- 验收标准

### 3.4 不知道为什么“会写文档”不等于“会写规格”

许多新人有文字表达能力，但缺少工程表达能力。

他们可能能把背景说得很完整，却不能把需求写成开发和测试能直接使用的形式。这正是本书要重点解决的问题。

## 4. 产品目标

这本书的总体目标，是帮助初学者建立一套完整、工程化的规格写作能力。

具体目标如下：

- 让读者理解规格在软件开发中的作用，而不是把规格当作文档格式
- 让读者分清 PRD、Spec、User Story、Acceptance Criteria 等概念的职责边界
- 让读者掌握从业务问题到可执行规格的收敛方法
- 让读者学会写功能需求、业务规则、状态变化、异常流程、边界条件和验收标准
- 让读者能用统一结构编写一套新人可落地的规格文档
- 让读者知道怎样评审规格、发现规格漏洞，并持续改进

## 5. 非目标

为了控制本书边界，以下内容不作为本书核心目标：

- 不把本书写成纯项目管理方法论教材
- 不把本书写成纯敏捷流程教材
- 不把本书写成某个特定公司内部模板大全
- 不深入展开数据库设计、系统架构设计、微服务治理等技术专题
- 不把重点放在某一种工具的使用方法上，例如某个协作文档平台如何操作
- 不以“背模板”为最终目标，而是强调思维方式和写作能力

这意味着本书会涉及这些内容与规格的关系，但不会把它们扩展成主线章节。

## 6. 目标读者

本书主要面向以下读者：

### 6.1 第一核心读者

- 刚进入软件行业的产品经理
- 刚开始参与需求定义的开发工程师
- 刚开始写测试场景和验收条件的测试工程师
- 需要快速建立规格意识的实习生和校招生

### 6.2 第二核心读者

- 已经写过一些需求文档，但总被反馈“写得不够落地”的初中级从业者
- 团队中需要统一规格写法和评审口径的人

### 6.3 不作为主要读者的人群

- 只想看某个模板快速交差、不关心背后逻辑的人
- 只关注高阶架构设计，不需要参与需求定义的人

## 7. 用户价值

读者看完这本书后，应该获得的核心价值不是“知道几个术语”，而是以下几种真实能力：

- 能判断一份文档是不是规格，而不仅仅是描述
- 能把模糊需求逐步追问成清晰需求
- 能识别缺失的角色、规则、边界、异常和验收条件
- 能独立写出一份结构完整、可评审、可开发、可测试的基础规格
- 能和产品、开发、测试围绕同一份规格进行有效沟通

## 8. 核心理念

本书在内容设计上要始终围绕以下核心理念：

### 8.1 规格不是文件，是约束

规格真正的价值不在于文件名，不在于格式，而在于它是否定义了系统行为和边界。

### 8.2 规格必须能落地

如果一段内容不能帮助开发实现、测试验证或团队评审，它就不是合格的工程规格内容。

### 8.3 例子优先于空泛描述

本书要尽量用真实的业务例子、错误示例、改写示例来讲解，而不是停留在抽象说理。

### 8.4 由浅入深，而不是一上来就堆术语

本书必须考虑新人阅读体验，先建立问题意识和判断标准，再逐步进入写作方法和结构细节。

## 9. 内容范围

本书应覆盖软件规格写作的核心链路，至少包括以下内容模块：

### 9.1 基础认知模块

- 什么是需求
- 什么是规格
- 为什么规格不是普通文档
- 规格在开发流程中的位置

### 9.2 概念区分模块

- PRD 的职责
- Spec 的职责
- User Story 的职责
- Acceptance Criteria 的职责
- 它们之间如何衔接

### 9.3 写作方法模块

- 如何定义问题
- 如何定义目标与范围
- 如何识别角色和场景
- 如何拆解功能需求
- 如何抽取业务规则
- 如何补齐异常流程和边界条件
- 如何写验收标准

### 9.4 结构模板模块

- PRD 结构
- 功能 Spec 结构
- 规则定义结构
- 验收标准结构

### 9.5 评审与改进模块

- 怎么评审一份规格
- 常见错误有哪些
- 如何从错误写法改成正确写法

### 9.6 综合案例模块

- 通过一个或多个完整案例，把从业务问题到规格成稿的过程串起来

## 10. 教程组织形式

本书不采用“一个大文件讲完所有内容”的形式，而采用分章节逐步展开的教程结构。

文件命名规则采用：

- `01 xxx.md`
- `02 xxx.md`
- `03 xxx.md`

这样做的目的有三点：

- 阅读顺序清晰，新人知道从哪里开始
- 每章主题单一，便于深入展开
- 以后补充或重构某一章时，不会破坏整本书结构

## 11. 章节规划

以下是建议的基础章节规划。后续可以在保持主线不变的前提下微调。

### 11.1 前置基础章节

- `01 什么是需求与规格.md`
  目标：建立最基础的判断标准，让读者知道“规格”到底在解决什么问题。

- `02 规格在软件开发流程里的位置.md`
  目标：解释规格与产品、设计、开发、测试、验收之间的关系。

- `03 常见需求文档的区别.md`
  目标：区分 PRD、Spec、User Story、Acceptance Criteria、测试用例等概念。

- `04 怎么把模糊想法变成可执行需求.md`
  目标：训练读者做需求追问和收敛。

- `05 规格写作的通用原则.md`
  目标：讲清楚清晰、完整、一致、可执行、可验证等原则。

### 11.2 核心写作章节

- `06 PRD 怎么写.md`
  目标：讲 PRD 的作用、结构、章节写法、示例、常见错误和模板。

- `07 User Story 怎么写.md`
  目标：讲用户故事的价值、边界和常见误用。

- `08 Acceptance Criteria 怎么写.md`
  目标：讲验收条件如何写成可测试表达。

- `09 功能 Spec 怎么写.md`
  目标：讲怎样定义功能行为、前置条件、主流程、分支流程和结果。

- `10 业务规则怎么写.md`
  目标：讲怎样沉淀规则而不是把规则散落在页面描述里。

- `11 状态、数据与边界怎么写.md`
  目标：讲状态流转、关键字段、异常与边界条件。

- `12 非功能需求怎么写.md`
  目标：讲性能、安全、稳定性、可观测性等要求。

### 11.3 提升与实践章节

- `13 怎么评审一份规格.md`
  目标：教读者如何发现规格中的漏洞。

- `14 常见错误与反例拆解.md`
  目标：用反例强化判断力。

- `15 从 PRD 到 Spec 到测试用例.md`
  目标：把整条链路串起来。

- `16 完整案例实战.md`
  目标：通过完整案例展示从问题定义到规格成文的全过程。

## 12. 内容深度要求

这本书不应停留在“列目录、给模板”的浅层教学。每一章都必须满足以下深度要求：

- 不只讲定义，还要讲为什么
- 不只讲正确写法，还要讲错误写法
- 不只讲结构，还要讲结构背后的工程目的
- 不只给规则，还要给例子
- 不只给正例，还要给反例和改写过程
- 不只讲主流程，还要提醒异常、边界和常见漏项

也就是说，本书的每一章都应该让新人不仅知道“写什么”，还知道“为什么这样写”和“不这样写会出什么问题”。

## 13. 写作风格要求

本书内容风格要统一，遵循以下原则：

- 语言尽量直接、清楚，不堆过度学术化表述
- 尽量减少空洞口号，增加工程语境里的解释
- 多使用业务例子，例如订单、退款、审批、搜索、导出、权限
- 每章最好包含“错误示例”和“改写示例”
- 对新人友好，避免默认读者已经有很强项目经验

## 14. 关键交付物

本项目的主要交付物包括：

- 一本完整的规格写作教程目录结构
- 一套按顺序编号的章节 Markdown 文件
- 一本书级别的 `prd.md`
- 每章明确的主题、目标、内容边界和示例支撑

最低交付标准不是“创建了很多文件”，而是形成一套可以连续阅读和学习的教程。

## 15. 验收标准

本书的需求完成，至少应满足以下验收条件：

### 15.1 结构验收

- 已存在一本书自己的 `prd.md`
- 教程按 `01`、`02`、`03` 这种方式组织
- 每一章主题明确，不互相混杂

### 15.2 内容验收

- 至少覆盖规格写作核心链路
- `PRD` 写法作为其中一章，而不是整本书的唯一内容
- 每章都应包含定义、作用、写法、示例、常见错误或注意事项

### 15.3 读者体验验收

- 新人从前几章开始阅读时，能够逐步建立理解，不会一开始就被术语压住
- 章节顺序符合学习路径，从认知到写法再到案例

### 15.4 工程性验收

- 内容应强调可执行、可验证、可评审，而不是停留在泛泛描述
- 至少有一章明确说明规格如何驱动开发和测试

## 16. 成功标准

本书写成后，理想状态下应达到以下效果：

- 新人能清楚说出 PRD、Spec、User Story、Acceptance Criteria 的区别
- 新人能独立指出一份需求文档的明显缺陷
- 新人能把一句模糊需求改写成带角色、规则、边界和验收条件的表达
- 新人能产出一份基础可用的规格文档初稿

如果读者看完后仍然只能背术语、不能动手写规格，那说明本书目标没有达成。

## 17. 风险与约束

本项目存在以下风险：

### 17.1 风险：内容过于抽象

如果只讲概念，不讲例子，新人很难吸收。

应对方式：

- 每章加入具体业务例子
- 多写错误写法与修正过程

### 17.2 风险：内容过于模板化

如果只给模板，读者容易机械套用，不理解背后逻辑。

应对方式：

- 每个模板都解释为什么这样设计
- 增加判断原则和取舍说明

### 17.3 风险：章节边界不清

如果 PRD、Spec、Story、AC 混写，新人会更混乱。

应对方式：

- 每章明确职责范围
- 在章节间反复强调它们的关系和区别

## 18. 后续编写原则

后续在正式写每一章时，应该遵守以下原则：

- 先保证学习路径正确，再追求内容扩展
- 先把基础章节写扎实，再往后写高级章节
- 每一章都要能独立成立，同时又服务于全书主线
- 章节之间要互相引用，但不能互相替代

## 19. 结论

这份 `prd.md` 的目的，是把这本书当成一个真正的产品来定义，而不是想到哪写到哪。

本书的核心产品定位可以总结为一句话：

`一套面向软件开发新人的、由浅入深的、强调工程落地的规格写作教程。`

而 `PRD 怎么写` 只是这套教程中的一个重要章节，不是整本书的全部。
