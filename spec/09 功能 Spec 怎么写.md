## 1. 为什么要专门讲功能 Spec

前面你已经学过：

- 什么是需求
- 什么是规格
- `PRD` 怎么写
- `User Story` 怎么写
- `Acceptance Criteria` 怎么写

但很多新人到这里仍然会遇到一个非常现实的问题：

- 我知道一个需求的大方向了
- 我也知道用户价值和验收标准该怎么想
- 可是一到“具体这个功能到底怎么写成规格”，还是不知道如何下手

这就是为什么要单独讲功能 `Spec`。

因为真正进入开发前，团队最需要的往往不是再多一层背景说明，而是：

- 这个功能到底怎么工作
- 谁可以用
- 在什么条件下触发
- 正常流程怎么走
- 异常怎么处理
- 状态怎么变化
- 什么规则不能违反

功能 `Spec` 就是把这些内容收敛成一份工程可执行定义。

## 2. 什么是功能 Spec

功能 `Spec`，可以简单理解为：

`对某一个具体功能的行为、规则、边界和结果做精确定义的规格说明。`

注意这里的重点有三个：

- `某一个具体功能`
- `行为和规则`
- `精确定义`

它不是需求全景总览，也不是高层目标描述，而是往具体功能上继续下沉。

例如：

- “优化运营查询效率”更像需求目标
- “订单列表支持按状态筛选，并定义权限、流程、空结果、异常行为”更像功能 `Spec`

### 2.1 功能 Spec 的本质

功能 `Spec` 的本质，是把一个功能写到足够明确，使得：

- 开发能实现
- 测试能验证
- 产品能评审
- 团队能对齐

### 2.2 它不是什么

它不是：

- 只有页面描述
- 只有目标描述
- 只有用户故事
- 只有技术实现细节

它是“行为定义文档”，不是“想法摘要”。

## 3. 功能 Spec 和 PRD 的区别

这是新人非常容易混的地方。

### 3.1 PRD 更偏全景

`PRD` 通常会覆盖：

- 背景
- 问题定义
- 目标
- 范围
- 用户角色
- 功能列表
- 验收标准

它回答的是：

- 这次为什么做
- 这次做什么
- 这次边界到哪里

### 3.2 功能 Spec 更偏局部精确

功能 `Spec` 则更聚焦：

- 某个具体功能怎么工作
- 哪些条件允许或禁止
- 触发后系统怎样响应
- 状态如何变化
- 异常如何处理

### 3.3 两者的关系

可以这样理解：

- `PRD` 定义需求全景
- 功能 `Spec` 把其中某一项功能继续写深

一个复杂需求可能只有一份 `PRD`，但会拆出多个功能 `Spec`。

例如：

- 退款系统改造 `PRD`
- 退款申请功能 `Spec`
- 客服审核功能 `Spec`
- 退款状态同步功能 `Spec`

## 4. 功能 Spec 到底要解决什么问题

功能 `Spec` 主要解决四类问题。

### 4.1 解决“开发不能靠猜”的问题

如果只有一句：

- 支持按状态筛选订单

开发仍然会继续问：

- 谁能筛
- 支持哪些状态
- 能不能多选
- 空结果怎么办
- 无权限怎么办

功能 `Spec` 的职责，就是把这些问题前置回答掉。

### 4.2 解决“测试不知道该覆盖什么”的问题

测试不是看见一个按钮就开始写用例，而是要依据规格判断：

- 主流程是什么
- 异常流程是什么
- 边界条件是什么

### 4.3 解决“产品和开发对细节理解不一致”的问题

很多返工不是因为开发能力不够，而是因为细节定义不清楚。

例如：

- 产品以为空结果要显示空状态页
- 开发以为返回空列表就够了

如果功能 `Spec` 没写，最后一定争议。

### 4.4 解决“复杂功能难以拆分”的问题

一个中等复杂度以上的功能，通常不可能只靠几句需求描述就稳定落地。

功能 `Spec` 可以帮助团队把复杂功能拆成：

- 角色
- 前置条件
- 主流程
- 分支流程
- 规则
- 异常
- 验收

这样开发和测试才有抓手。

## 5. 一份功能 Spec 最少应该写到什么程度

这个问题非常实际。

功能 `Spec` 不是越长越好，但至少要写到：

- 团队不用大量靠口头补充
- 开发不用大量靠猜
- 测试能设计关键用例

通常来说，一份基础合格的功能 `Spec`，至少应该交代清楚下面这些内容：

1. 这个功能是什么
2. 它为谁服务
3. 触发前提是什么
4. 用户如何触发
5. 主流程怎么走
6. 关键分支和异常怎么处理
7. 关键业务规则是什么
8. 涉及哪些状态变化
9. 怎样验收

如果这些问题大半都没有答案，这份 `Spec` 基本还不够。

## 6. 功能 Spec 的推荐结构

下面给一套非常实用的结构。

1. 功能概述
2. 目标角色
3. 前置条件
4. 触发方式
5. 主流程
6. 分支流程
7. 业务规则
8. 状态变化
9. 异常与边界条件
10. 非功能要求
11. 验收标准

这个结构不是唯一标准，但非常适合作为新人写功能规格的基础骨架。

## 7. 每一部分具体应该写什么

这一节是整章核心。

### 7.1 功能概述

这一节用来回答：

- 这个功能到底是什么
- 要解决哪个具体问题

不要写成长背景说明，而要简洁明确。

例如：

订单列表新增状态筛选能力，用于帮助运营和客服快速定位特定状态订单，减少人工翻页查找成本。

### 7.2 目标角色

写清楚：

- 谁可以使用
- 谁可能被影响
- 权限边界是否不同

例如：

- 运营人员：可使用状态筛选查看权限范围内订单
- 客服人员：可使用状态筛选查看权限范围内订单
- 无订单查看权限用户：不可使用该功能

### 7.3 前置条件

写清楚这个功能成立之前，系统或用户必须满足什么条件。

例如：

- 用户已登录后台
- 用户具有订单查看权限
- 当前位于订单列表页

前置条件写清楚，可以减少很多讨论歧义。

### 7.4 触发方式

写清楚功能是如何被触发的。

例如：

- 用户选择订单状态并点击“搜索”
- 用户点击“申请退款”按钮
- 定时任务到达同步时间点

不同触发方式对应不同规格写法。不要笼统写“系统执行该功能”。

### 7.5 主流程

主流程写的是“理想且正常的主要成功路径”。

例如：

1. 用户进入订单列表页
2. 选择订单状态“待审核”
3. 点击“搜索”
4. 系统按状态过滤并返回订单列表
5. 页面展示查询结果

主流程应该清楚、按顺序，并且尽量避免跳步。

### 7.6 分支流程

分支流程写的是：

- 正常场景中的变体
- 非主路径但可预期的情况

例如：

- 用户未选择状态时，系统按其他筛选条件执行查询
- 用户切换为其他状态后重新搜索，系统刷新为新结果

分支流程和异常流程不要混为一谈。

### 7.7 业务规则

业务规则是功能 `Spec` 最重要的一部分之一。

例如：

- 仅有订单查看权限的角色可使用该筛选功能
- 状态筛选仅支持单选
- 用户仅可查看其权限范围内的订单

如果规则不单独写清楚，往往会散落在流程描述里，最后很难维护。

### 7.8 状态变化

不是所有功能都有明显状态流转，但只要功能会影响系统状态，就应该明确写出。

例如在退款申请功能里：

- 提交前：退款状态为“无”
- 提交后：退款状态变为“申请中”

如果是纯查询类功能，也可以说明：

- 本功能不改变订单状态，只影响列表展示结果

### 7.9 异常与边界条件

这是新手最容易漏的部分。

例如：

- 无权限用户访问怎么办
- 查询结果为空怎么办
- 状态值无效怎么办
- 接口超时怎么办

这部分不写，开发和测试都容易靠猜。

### 7.10 非功能要求

例如：

- 查询接口在 95% 情况下应于 2 秒内返回
- 关键查询操作应记录审计日志

不是每个功能都要写很多，但关键功能最好明确。

### 7.11 验收标准

功能 `Spec` 最后一定要收回到：

- 做到什么程度算满足

否则规格定义没有闭环。

## 8. 如何写主流程

主流程看起来简单，实际上也有很多写法差异。

### 8.1 主流程写作目标

主流程的目标不是讲故事，而是把关键动作链条写清楚。

### 8.2 主流程写作建议

- 按时间顺序写
- 一步只写一个主要动作
- 写清用户动作和系统响应
- 不要混进太多异常说明

### 8.3 不推荐写法

例如：

- 用户进入页面后进行操作，系统返回结果

这句话信息太少，等于没写。

### 8.4 推荐写法

例如：

1. 用户进入订单列表页
2. 用户选择状态“待审核”
3. 用户点击“搜索”
4. 系统校验用户查看权限
5. 系统返回状态为“待审核”的订单列表
6. 页面展示查询结果

这样的主流程，开发和测试都更容易使用。

## 9. 如何写分支流程

很多人会漏掉分支流程，或者把分支和异常全混在一起。

### 9.1 什么是分支流程

分支流程通常是：

- 在主流程基础上的合法变化路径

例如：

- 用户不选状态也能搜索
- 用户切换状态重新搜索
- 用户同时输入订单号和状态组合查询

### 9.2 分支流程为什么重要

因为真实用户操作很少只走唯一主路径。

如果不写分支，最后这些路径通常会在开发或测试时临时补，容易出错。

### 9.3 分支和异常的区别

- 分支：合法变化路径
- 异常：出错、无权限、非法状态、失败场景

这两个最好分开写。

## 10. 如何写业务规则

功能 `Spec` 写不好，往往不是流程不会写，而是规则没写透。

### 10.1 业务规则应该长什么样

推荐写成明确判断句。

例如：

- BR-1：仅有订单查看权限的用户可使用订单状态筛选功能
- BR-2：状态筛选仅支持单选
- BR-3：用户仅可查看其权限范围内订单

### 10.2 为什么规则要独立编号

因为规则通常会：

- 被多个流程复用
- 被测试直接引用
- 在评审时被重点讨论

独立编号能极大提升可维护性。

### 10.3 规则不要藏在说明性段落里

错误写法常见于：

- 文中零散提一下
- 写在某个例子里
- 只有页面说明，没有明确规则

规则必须显式定义。

## 11. 如何写状态变化

只要一个功能会影响系统状态，就必须显式说明。

### 11.1 为什么状态变化重要

因为很多 bug 都不是“页面没显示”，而是：

- 状态改错了
- 状态在不该变时变了
- 状态没同步

### 11.2 状态变化的常见写法

可以写成：

- 操作前状态
- 触发事件
- 操作后状态

例如：

- 当用户提交退款申请时，退款状态由“无”变为“申请中”

### 11.3 查询类功能也可以说明“不改变状态”

这同样是一种重要定义。

例如：

- 本功能仅影响订单列表展示结果，不改变任何订单业务状态

## 12. 如何写异常和边界条件

这一节是功能 `Spec` 是否真正像工程文档的关键分水岭。

### 12.1 至少要考虑哪些异常

- 权限不足
- 输入非法
- 状态不允许
- 数据为空
- 外部接口失败
- 请求超时
- 重复操作

### 12.2 示例

- EC-1：无订单查看权限的用户访问列表页时，系统应提示无权限，且不展示订单数据
- EC-2：用户输入的状态值无效时，系统应提示筛选条件无效，并阻止查询
- EC-3：查询结果为空时，页面应展示空状态提示
- EC-4：订单服务 5 秒内未返回结果时，系统应提示查询超时，并允许用户重试

### 12.3 边界条件也不能漏

边界包括：

- 空值
- 最小值 / 最大值
- 状态边界
- 权限边界
- 时间边界

这部分如果规格不写，测试很难系统性覆盖。

## 13. 如何和 Acceptance Criteria 衔接

很多新人写完功能 `Spec` 后，不知道怎么落到验收。

最简单的理解是：

- 功能 `Spec` 负责定义
- `Acceptance Criteria` 负责定义哪些结果必须被验证通过

### 13.1 从哪里提炼验收标准

通常从这些地方提炼：

- 主流程中的核心成功结果
- 关键业务规则
- 关键异常处理
- 关键非功能要求

### 13.2 一个简单例子

功能 `Spec` 写：

- 用户选择“待审核”状态并点击搜索后，系统返回待审核订单

可以提炼成验收标准：

- Given 用户已登录且具有订单查看权限
- When 用户选择“待审核”状态并点击搜索
- Then 系统应仅返回待审核订单

### 13.3 不要把所有 Spec 内容都原样搬成验收标准

验收标准要抓关键结果，不是把整份规格再抄一遍。

## 14. 一个完整例子：订单状态筛选功能 Spec

下面给一个简化但完整的例子。

### 14.1 功能概述

订单列表新增状态筛选能力，用于帮助运营和客服快速定位目标状态订单。

### 14.2 目标角色

- 运营人员
- 客服人员

### 14.3 前置条件

- 用户已登录后台
- 用户具有订单查看权限
- 当前位于订单列表页

### 14.4 触发方式

- 用户选择一个订单状态并点击“搜索”

### 14.5 主流程

1. 用户进入订单列表页
2. 用户选择订单状态“待审核”
3. 用户点击“搜索”
4. 系统校验用户订单查看权限
5. 系统按所选状态过滤订单列表
6. 页面展示筛选结果

### 14.6 分支流程

1. 用户未选择状态时，系统按其他已有筛选条件执行查询
2. 用户修改状态后再次点击搜索，系统返回新的查询结果

### 14.7 业务规则

- BR-1：仅有订单查看权限的用户可使用该功能
- BR-2：状态筛选仅支持单选
- BR-3：用户仅可查看其权限范围内订单

### 14.8 状态变化

- 本功能不改变订单业务状态，仅影响列表展示结果

### 14.9 异常与边界条件

- EC-1：无权限用户访问时，系统提示无权限且不展示订单数据
- EC-2：无效状态值时，系统提示筛选条件无效并阻止查询
- EC-3：查询结果为空时，页面展示空状态提示
- EC-4：查询超时时，系统提示查询超时并允许重试

### 14.10 非功能要求

- NFR-1：查询请求在 95% 情况下应于 2 秒内返回

### 14.11 验收标准

- 当有权限用户选择“待审核”状态并点击搜索时，系统应仅返回待审核订单
- 当无权限用户访问订单列表页时，系统应提示无权限，且不展示订单数据
- 当查询结果为空时，系统应展示空状态提示

这个例子已经足够让开发和测试开始工作，而不需要大量脑补。

## 15. 常见错误写法

### 15.1 错误一：只有页面描述，没有行为定义

例如：

- 页面上放一个状态下拉框和一个搜索按钮

这不是功能 `Spec`，这只是界面说明。

### 15.2 错误二：只有目标，没有规则

例如：

- 支持运营快速筛选订单

这还是停留在需求目标，不是功能规格。

### 15.3 错误三：只写主流程，不写异常

这种写法非常常见，也非常危险。

### 15.4 错误四：把技术方案写进功能 Spec

例如：

- 后端使用某缓存方案
- 前端使用某组件库

这些通常不属于功能 `Spec` 的核心内容。

### 15.5 错误五：没有验收闭环

写完很多规则和流程，但最后没有明确怎样算完成。

这会导致规格缺少最后一层落地标准。


这个模板不是为了让你机械套，而是提醒你：

- 功能规格最容易漏的维度有哪些

## 17. 这一章你应该带走什么

读完这一章，你至少应该记住下面几点。

第一，功能 `Spec` 是把某个具体功能写成可开发、可测试、可评审的行为定义。

第二，它和 `PRD` 的区别在于：

- `PRD` 更偏需求全景
- 功能 `Spec` 更偏局部精确定义

第三，一份基础合格的功能 `Spec`，至少应覆盖：

- 功能概述
- 角色
- 前置条件
- 触发方式
- 主流程
- 分支流程
- 规则
- 状态变化
- 异常与边界
- 验收标准

第四，功能 `Spec` 最常见的问题不是写得太少，而是：

- 只有目标没有规则
- 只有页面没有行为
- 只有主流程没有异常

第五，写完功能 `Spec` 后，一定要能继续回答：

- 开发能不能实现
- 测试能不能验证
- 产品能不能验收

如果这三个问题还回答不了，规格通常就还不够。

## 小结

这一章最核心的判断是：

`功能 Spec 的价值，不在于把一个功能说得更长，而在于把它定义得更清楚、更可执行。`

后面继续写规则、状态、边界这些章节时，你会发现它们本质上都是在强化功能 `Spec` 的质量。
