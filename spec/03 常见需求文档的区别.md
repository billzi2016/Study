## 1. 为什么新人总会把这些文档混在一起

很多新人刚接触软件开发文档时，会很自然地产生一种感觉：

- 好像全都在写“需求”
- 好像全都和功能有关
- 好像只是名字不同，格式不同

于是很容易出现下面这些情况：

- 把一句 `User Story` 当成完整需求
- 把 `PRD` 写成页面说明书
- 把 `Spec` 写成愿望清单
- 把 `Acceptance Criteria` 写成口号
- 把测试用例当成规格本身

这很正常，因为这些东西本来就经常出现在同一个项目里，而且它们彼此相关。

但“彼此相关”不等于“彼此等价”。

如果你一直分不清这些文档和表达方式的职责边界，后面很容易出现两个问题：

- 文档写了很多，但不落地
- 项目看起来流程完整，但信息在传递过程中一直失真

所以这一章的核心任务，就是帮你把这些常见东西按层次拆开。

## 2. 先建立一个总判断：这些东西不是同一层

先给你一个最重要的总判断：

`PRD、Spec、User Story、Use Case、Acceptance Criteria、Test Case，不是在同一层解决问题。`

它们关注的重点不同。

可以先粗略记成这样：

- `PRD`：讲这次为什么做、做什么、范围是什么
- `Spec`：讲系统具体应该怎样行为
- `User Story`：讲用户想获得什么价值
- `Use Case`：讲某个使用场景中的交互过程
- `Acceptance Criteria`：讲怎样判断这项需求完成了
- `Test Case`：讲测试怎么验证它

你可以把它们想成一条链路上的不同位置，而不是同一种东西的不同叫法。

## 3. PRD 是什么，解决什么问题

`PRD` 通常指 Product Requirements Document，也就是产品需求文档。

它主要回答的是：

- 为什么做这件事
- 这件事要解决什么问题
- 面向哪些用户
- 本期范围是什么
- 目标是什么
- 非目标是什么
- 主要功能和规则有哪些

### 3.1 PRD 的主要职责

PRD 更像一份需求全景定义。

它的作用是把一个业务问题整理成团队可讨论、可评审、可推进的需求整体。

它通常会包含：

- 背景
- 问题定义
- 目标
- 范围
- 用户角色
- 功能需求
- 业务规则
- 非功能要求
- 验收标准

### 3.2 PRD 最擅长解决什么问题

PRD 最擅长解决的是：

- 团队这次到底在做什么
- 为什么值得做
- 这次边界到哪里

换句话说，PRD 最重要的价值是帮助团队对“需求整体”形成共识。

### 3.3 PRD 不擅长单独完成什么

虽然很多团队把大量细节也写进 PRD，但 PRD 本身不是万能的。

如果某个复杂功能需要精确到：

- 每个状态变化
- 每个输入输出
- 每种异常处理
- 每个接口字段约束

那么通常还需要更细的规格表达，而不能只靠一句“PRD 里写了”。

### 3.4 一句话记忆

`PRD` 更偏“这次需求整体怎么定义”。

## 4. Spec 是什么，解决什么问题

`Spec` 是规格，重点不是文档名，而是定义方式。

它关注的是：

- 系统应该如何行为
- 行为边界是什么
- 输入输出是什么
- 规则是什么
- 异常怎么处理
- 什么样算符合要求

### 4.1 Spec 的主要职责

Spec 最核心的职责，是把需求收敛成可执行、可验证的行为定义。

它要尽量减少这些问题：

- 开发各自理解
- 测试靠猜
- 产品靠口头补充
- 上线后才发现规则没说清

### 4.2 Spec 更像工程约束

PRD 更多是在定义这次需求全景，Spec 更像是在定义这次行为契约。

例如：

- PRD 说：支持运营人员快速定位待审核订单
- Spec 会继续写：哪些角色可用、如何筛选、无权限怎么办、空结果怎么办、性能要求是什么

### 4.3 Spec 不一定单独存在为一个文件

这一点很重要。

很多新人会误以为：

- 只有名字叫 `spec.md` 的文件才叫规格

这不对。

规格是一种内容属性，不是一个固定文件名。

也就是说：

- `PRD` 里可以有规格内容
- 接口文档里可以有规格内容
- 测试前置文档里也可能有规格内容

关键不在于文件名，而在于内容是否在定义行为和约束。

### 4.4 一句话记忆

`Spec` 更偏“系统具体该怎么做，边界和规则是什么”。

## 5. User Story 是什么，解决什么问题

`User Story` 是用户故事。

它常见写法是：

`作为 <某类用户>，我想要 <某个能力>，从而 <获得某种价值>。`

例如：

`作为运营人员，我想按订单状态筛选订单，从而快速定位待审核订单。`

### 5.1 User Story 的主要职责

User Story 最重要的作用，不是提供完整规格，而是提醒团队：

- 需求是为谁服务的
- 用户到底想完成什么
- 这项能力能带来什么价值

也就是说，它把视角放回到用户和价值，而不是一上来就沉到实现细节。

### 5.2 User Story 能解决什么问题

它最适合解决：

- 避免团队为了功能而功能
- 在讨论中持续对齐用户价值
- 用简短形式表达一个需求方向

### 5.3 User Story 的局限

User Story 通常不够开发，也不够测试。

例如这句话：

- 作为运营人员，我想按订单状态筛选订单，从而快速定位待审核订单。

它还没回答：

- 支持哪些状态
- 能否多选
- 默认值是什么
- 权限怎么控制
- 空结果怎么处理

所以 User Story 是起点，不是完整规格。

### 5.4 一句话记忆

`User Story` 更偏“从用户视角表达需求价值”。

## 6. Use Case 是什么，解决什么问题

`Use Case` 通常翻译为用例或使用案例。

它关注的是：

- 某个角色在某个场景里
- 为了完成某个目标
- 和系统之间发生了怎样的交互过程

### 6.1 Use Case 的主要职责

它适合描述一个场景中的过程。

常见会写：

- 参与者
- 前置条件
- 触发条件
- 主成功场景
- 扩展场景
- 异常场景

### 6.2 Use Case 擅长解决什么问题

它特别适合帮助团队理解：

- 用户一步一步是怎么完成任务的
- 系统在哪些节点做出响应
- 主流程和异常流程分别是什么

### 6.3 Use Case 和 User Story 的区别

很多人会把这两个混掉。

区别很简单：

- `User Story` 更短，更偏价值表达
- `Use Case` 更展开，更偏场景过程

例如：

User Story：

- 作为用户，我想在线申请退款，从而减少联系客服等待时间。

Use Case：

- 用户进入订单详情页
- 点击申请退款
- 选择退款原因并提交
- 系统校验订单状态
- 系统创建退款申请并更新状态

### 6.4 一句话记忆

`Use Case` 更偏“某个场景里的交互过程怎么发生”。

## 7. Acceptance Criteria 是什么，解决什么问题

`Acceptance Criteria` 是验收标准。

它回答的是：

- 这项需求做到什么程度，才算完成
- 哪些行为是必须满足的
- 测试该按什么标准验证

### 7.1 Acceptance Criteria 的主要职责

它的主要职责是把“完成定义”写清楚。

也就是说，它不是再讲一次背景，不是再讲一次目标，而是在说：

- 满足哪些条件，算交付通过

### 7.2 为什么它特别重要

因为很多项目争议都发生在最后：

- 产品说没做完
- 开发说已经做了
- 测试说不知道按什么判

如果没有明确验收标准，大家对“完成”的理解会完全不同。

### 7.3 常见写法

常见写法包括：

- 列表式条件
- Given / When / Then

例如：

- Given 运营人员已登录订单后台
- When 其选择“待审核”状态并点击搜索
- Then 系统应仅返回待审核订单

### 7.4 Acceptance Criteria 和 Spec 的关系

验收标准是规格的重要组成部分，但不等于完整规格。

规格会讲：

- 规则
- 流程
- 异常
- 边界

验收标准则更聚焦：

- 最终怎样判断是否满足要求

### 7.5 一句话记忆

`Acceptance Criteria` 更偏“怎样算做完、怎样算符合要求”。

## 8. Test Case 是什么，解决什么问题

`Test Case` 是测试用例。

它解决的问题是：

- 测试人员怎样具体验证系统行为

### 8.1 Test Case 的主要职责

测试用例通常会写得更操作化，例如：

- 前置条件
- 测试步骤
- 输入数据
- 预期结果

### 8.2 Test Case 和 Acceptance Criteria 的区别

很多新人会把这两个直接画等号，但其实不一样。

Acceptance Criteria 更像：

- 这项需求必须满足哪些结果

Test Case 更像：

- 我准备怎样一步一步验证这些结果

也就是说：

- 验收标准定义验证目标
- 测试用例定义验证动作

### 8.3 一句话记忆

`Test Case` 更偏“测试具体怎么验”。

## 9. 它们之间到底是什么关系

这几个概念可以放进一条从上到下的链路里理解。

一个相对实用的理解方式是：

`业务问题 -> 需求定义 -> PRD -> User Story -> Spec -> Acceptance Criteria -> Test Case -> 开发实现与测试执行`

这里要注意两点。

### 9.1 这不是严格线性流水线

真实工作中，这些内容经常会反复迭代，而不是一次写完一路往下。

比如：

- 写规格时发现需求范围不合理
- 写验收标准时发现业务规则漏了
- 写测试用例时发现异常场景没定义

这都很正常。

### 9.2 它们不是互相替代，而是互相补位

例如：

- 只有 User Story，不够开发
- 只有技术方案，不够定义业务目标
- 只有测试用例，不够表达需求背景

所以不要想着“用一种东西替代全部”，而要明白它们各自负责哪一层。

## 10. 一个完整例子：同一个需求在不同文档里分别怎么写

这一节非常重要，因为很多人真正学会区分，靠的不是定义，而是并排对比。

我们用同一个需求来演示：

- 订单后台支持按订单状态筛选订单

### 10.1 业务问题

运营人员在处理异常订单时，当前只能按订单编号搜索，无法快速定位待审核订单，导致处理效率低。

### 10.2 PRD 写法

背景：

- 当前订单后台缺少状态筛选能力，运营需要人工翻页查找待审核订单，效率低。

目标：

- 支持按订单状态筛选订单
- 缩短运营定位待审核订单的时间

范围：

- 本期支持单状态筛选
- 本期不支持多状态组合筛选和导出

### 10.3 User Story 写法

- 作为运营人员，我想按订单状态筛选订单，从而快速定位待审核订单。

### 10.4 Use Case 写法

参与者：

- 运营人员

前置条件：

- 已登录后台并具有订单查看权限

主流程：

1. 运营人员进入订单列表页
2. 选择订单状态“待审核”
3. 点击搜索
4. 系统返回待审核订单列表

扩展场景：

1. 若结果为空，系统展示空状态
2. 若无权限，系统提示无权限

### 10.5 Spec 写法

- 订单列表页新增订单状态筛选控件。
- 具有订单查看权限的运营人员和客服人员可使用该筛选功能。
- 筛选仅支持单选。
- 用户选择状态并点击搜索后，系统仅返回符合该状态的订单。
- 当结果为空时，系统展示空状态提示。
- 当用户无权限时，不展示订单数据并提示无权限。
- 查询请求在 95% 情况下应于 2 秒内返回。

### 10.6 Acceptance Criteria 写法

- Given 用户已登录且具有订单查看权限
- When 用户选择“待审核”状态并点击搜索
- Then 系统应仅返回待审核订单

- Given 用户无订单查看权限
- When 用户访问订单列表页
- Then 系统应提示无权限，且不展示订单数据

### 10.7 Test Case 写法

测试用例 1：

- 前置条件：使用具有订单查看权限的运营账号登录
- 操作步骤：
  1. 进入订单列表页
  2. 选择状态“待审核”
  3. 点击搜索
- 预期结果：
  1. 页面仅展示待审核订单
  2. 查询成功返回

测试用例 2：

- 前置条件：使用无权限账号登录
- 操作步骤：
  1. 进入订单列表页
- 预期结果：
  1. 页面提示无权限
  2. 不展示订单数据

看完这组对比，你应该能明显感觉到：

- 它们围绕的是同一个需求
- 但每一种表达解决的问题都不同

## 11. 新人最常见的混用方式

### 11.1 用 User Story 代替完整需求

错误想法：

- 我已经写了“作为谁，我想要什么，从而得到什么”，所以需求写完了

问题在于，这只表达了价值方向，没有表达规则和边界。

### 11.2 用页面描述代替 Spec

错误写法：

- 页面上有个输入框和按钮

这描述的是界面，不是规格。

### 11.3 用 Acceptance Criteria 代替全部规格

验收标准很重要，但它不负责完整讲清：

- 所有业务规则
- 所有状态变化
- 所有场景约束

### 11.4 用 Test Case 回头补需求漏洞

测试用例可以帮你发现漏洞，但不适合作为需求定义的起点。

如果到了测试阶段才第一次补齐规则，代价通常会更高。

### 11.5 把 PRD 写成技术方案

例如：

- 接口采用某种架构
- 数据库采用某种索引

这些内容不是 PRD 的核心职责。

## 12. 实际工作里怎么判断“现在该写哪一种”

这是很实用的一节。

你可以直接问自己下面这些问题。

### 12.1 你是在解释“为什么做”和“做什么”吗

如果是，优先考虑 `PRD` 或需求定义内容。

### 12.2 你是在表达“用户想获得什么价值”吗

如果是，优先考虑 `User Story`。

### 12.3 你是在描述“用户和系统之间如何一步步交互”吗

如果是，优先考虑 `Use Case`。

### 12.4 你是在定义“系统到底该怎样行为、边界是什么”吗

如果是，优先考虑 `Spec`。

### 12.5 你是在定义“做到什么程度算完成”吗

如果是，优先考虑 `Acceptance Criteria`。

### 12.6 你是在定义“测试时具体怎么操作验证”吗

如果是，优先考虑 `Test Case`。

这个判断方法的重点不是背术语，而是先看你正在解决什么问题。

## 13. 这一章你应该带走什么

读完这一章，你至少应该明确下面几点。

第一，`PRD`、`Spec`、`User Story`、`Use Case`、`Acceptance Criteria`、`Test Case` 不是一回事，它们解决的是不同层的问题。

第二，最容易记的一组区分是：

- `PRD` 讲需求整体
- `Spec` 讲行为约束
- `User Story` 讲用户价值
- `Use Case` 讲场景过程
- `Acceptance Criteria` 讲完成标准
- `Test Case` 讲验证步骤

第三，它们通常不会孤立出现，而是围绕同一个需求互相补位。

第四，如果你把这些东西混用，最后最常见的结果就是：

- 文档看起来很多
- 但真正能指导开发和测试的内容很少

第五，后面你再继续学习每种具体写法时，要始终记住一个判断：

`先问自己：我现在要解决的问题到底是哪一层的问题？`

只要这一步清楚，你写出来的文档就会越来越稳。

## 小结

这一章不是让你死记术语，而是让你建立职责边界意识。

你只要真正理解了这些表达方式各自解决什么问题，后面不管团队用什么模板、什么工具、什么命名方式，你都不容易再被表面形式带偏。
