## 1. 什么是 PRD

PRD，通常指 Product Requirements Document，也就是产品需求文档。

在软件开发里，PRD 的作用不是“把想法写下来”，而是把一个业务问题，转换成一份团队可以共同理解、共同讨论、共同执行的需求定义。

一份好的 PRD，至少要做到四件事：

- 让产品知道这次到底要解决什么问题。
- 让设计知道要支撑哪些行为和信息结构。
- 让开发知道系统需要做什么、不需要做什么。
- 让测试知道应该如何验证这次交付是否合格。

所以，PRD 的本质不是说明书，不是汇报材料，也不是老板看了觉得“写得很多”的文档。PRD 的本质是：

`把模糊的业务目标，收敛成可以讨论、可以拆解、可以开发、可以测试、可以验收的需求约束。`

如果一份 PRD 写完以后：

- 开发还是不知道边界在哪
- 测试还是不知道该怎么验
- 评审时每个人理解都不一样
- 做到一半还在不停补规则

那这份 PRD 大概率不合格。

## 2. PRD 不是什么

很多新人一开始写 PRD，最容易犯的错，就是把 PRD 写成以下几种东西。

### 2.1 PRD 不是“功能愿望清单”

错误写法：

- 增加搜索
- 优化列表
- 支持导出
- 提升体验

这种写法的问题是，只有结论，没有定义。团队根本不知道：

- 搜什么
- 谁能搜
- 搜哪些字段
- 支持模糊还是精确
- 导出什么格式
- 数据范围多大
- 有没有权限限制

“想做什么”不是需求，“做成什么样算完成”才接近需求。

### 2.2 PRD 不是“页面描述稿”

错误写法：

- 页面上面有一个按钮
- 按钮旁边有一个输入框
- 下面是一个表格

这类写法只描述了界面表象，没有描述业务行为。开发真正需要的是：

- 这个按钮做什么
- 点击后触发什么校验
- 成功和失败分别怎么处理
- 是否涉及权限控制
- 是否影响其他状态

页面结构只是表现形式，需求本身应该定义行为和规则。

### 2.3 PRD 不是“技术方案文档”

PRD 不应该直接规定：

- 数据库怎么建表
- API 用 REST 还是 GraphQL
- 前端用什么组件库
- 缓存放 Redis 还是本地内存

这些是技术方案、设计方案、实现方案的内容。

PRD 可以提出约束，例如：

- 查询结果需要在 2 秒内返回
- 审计日志必须保留 180 天
- 导出文件最大支持 10 万行

但不应该越权定义具体技术实现，除非这是组织内明确约定的写法。

### 2.4 PRD 不是“拍脑袋后的口语记录”

错误写法常见于会议纪要式 PRD：

- 这里先简单做一下
- 后面看情况再补
- 这个逻辑和之前差不多
- 用户应该懂

这种表述最大的问题是不可验证。没有人能依据“差不多”“先简单做”来开发。

文档一旦进入工程流程，就必须从口语切换为可定义、可验证、可追责的表达。

## 3. PRD、Spec、User Story、Acceptance Criteria 的关系

很多团队会把这些词混着用，但它们不是一回事。

### 3.1 PRD 是需求全景

PRD 关注的是“这次要解决什么问题，以及需求整体是什么”。

PRD 更偏全局视角，通常包括：

- 背景
- 目标
- 范围
- 用户
- 功能
- 规则
- 非功能要求
- 验收方式

PRD 回答的是：`为什么做、做什么、做到什么程度。`

### 3.2 Spec 是更精确的规格定义

Spec 更强调精确、约束、可验证。它会更进一步回答：

- 某个行为在什么前提下发生
- 输入是什么
- 输出是什么
- 边界是什么
- 异常怎么处理
- 如何验收

如果说 PRD 是“需求层面的定义”，那么 Spec 更接近“工程执行层面的定义”。

很多成熟团队会从 PRD 继续往下收敛出：

- 功能规格说明
- 接口规格
- 状态流转规格
- 验收规格
- 测试场景

### 3.3 User Story 是用户价值表达

User Story 常见格式：

`作为 <某类用户>，我想要 <某个能力>，从而 <获得某种价值>。`

例如：

`作为运营人员，我想按订单状态筛选订单，从而快速定位待处理订单。`

User Story 的价值在于提醒团队：需求不是为了功能而功能，而是为了用户价值。

但 User Story 本身不够开发。因为它通常没有写清：

- 筛选哪些状态
- 能否多选
- 默认值是什么
- 权限限制是什么
- 空结果怎么展示

所以 User Story 是起点，不是终点。

### 3.4 Acceptance Criteria 是验收条件

Acceptance Criteria 是“这个需求完成没有”的判断标准。

它必须做到：

- 清晰
- 具体
- 可测试
- 无歧义

例如：

- Given 运营人员已登录订单后台
- When 其在订单列表选择“待支付”状态并点击搜索
- Then 系统仅返回状态为“待支付”的订单
- And 搜索结果应在 2 秒内展示

Acceptance Criteria 是连接需求和测试的关键桥梁。

### 3.5 它们之间的关系

可以把它们理解成一条收敛链路：

`业务目标 -> PRD -> User Story -> Spec -> Acceptance Criteria -> 测试用例 -> 开发实现`

不是每个团队都把这些文档分得很细，但思维上一定要分清楚。否则新人很容易把一句用户故事，当成完整需求；或者把一个页面草图，当成可开发规格。

## 4. 为什么很多团队写了 PRD 还是做不好项目

根本原因通常不是“没写文档”，而是“文档没有起到约束作用”。

常见问题有五类。

### 4.1 只有目标，没有规则

例如：

- 支持用户申请退款

这句话只说了要做什么，没有说：

- 谁可以申请
- 订单什么状态可以申请
- 退款是否支持部分退款
- 审核流程是否人工参与
- 退款失败是否可重试

这会导致开发各自理解，测试也无法覆盖完整场景。

### 4.2 只有主流程，没有异常流程

很多 PRD 只写“正常人正常使用”的情况，却不写：

- 网络失败
- 数据为空
- 权限不足
- 重复点击
- 状态过期
- 外部接口超时

真实系统最容易出事故的地方，恰恰是在这些“非理想情况”里。

### 4.3 把“想法”当“需求”

例如：

- 搜索要更智能
- 页面要更清晰
- 流程要更顺滑

这类话可以出现在讨论里，但不能直接作为 PRD 落地。因为没有明确标准。

需求必须能回答一个问题：`别人按文档做，能不能做出同一件事？`

如果不能，就说明定义不够。

### 4.4 范围没有封边

一个 PRD 最怕“什么都想顺手做一下”。

比如原需求只是优化筛选，结果评审中又加上：

- 顺便支持导出
- 顺便改下权限
- 顺便把详情页也重构

如果 PRD 没有明确写“本期范围内 / 本期范围外”，项目几乎一定会膨胀。

### 4.5 验收标准缺失

没有验收标准，就没有客观完成定义。

团队最后会陷入：

- 产品觉得没做完
- 开发觉得已经做了
- 测试不知道按什么标准提单

所以 PRD 最后一定要收敛到“怎么判定完成”。

## 5. 一份合格 PRD 的核心标准

可以用下面七个标准衡量一份 PRD。

### 5.1 正确

需求要真实反映业务问题，不能建立在错误理解上。

### 5.2 清晰

表达不能模糊，尽量避免：

- 尽量
- 合理
- 适当
- 优化
- 更好
- 类似之前

这些词如果没有定义，就没有价值。

### 5.3 完整

至少要覆盖：

- 背景
- 目标
- 范围
- 功能
- 规则
- 异常
- 验收

### 5.4 一致

前后定义不能冲突。例如前面写“仅管理员可操作”，后面示例却让普通用户也能操作，这就是不一致。

### 5.5 可执行

开发拿到后能拆任务，测试拿到后能写用例，设计拿到后能出稿。

### 5.6 可验证

每条关键需求都能判断“满足”还是“不满足”。

### 5.7 可维护

文档结构要清楚，编号要规范，后续变更时能知道改了哪里、影响了什么。

## 6. 写 PRD 之前先想清楚什么

很多人一上来就写页面和按钮，这是顺序错了。

真正好的写法应该从问题定义开始。

在动笔前，至少先回答下面几个问题。

### 6.1 这个需求解决的到底是什么问题

不要写“我们要做一个导出功能”，而要先回答：

- 现在为什么需要导出
- 哪类用户需要
- 他们目前怎么做
- 当前方式有什么痛点
- 这个痛点是否足够重要

### 6.2 谁是主要用户

不同用户看到的是完全不同的系统。

例如：

- 终端消费者关注操作简单
- 运营人员关注筛选效率
- 管理员关注权限和审计

如果不先分清用户，需求会写得非常混乱。

### 6.3 这个需求成功的标准是什么

比如：

- 客服平均处理时长从 8 分钟降到 5 分钟
- 退款申请误操作率降低 50%
- 查询接口 P95 响应低于 2 秒

如果没有成功标准，就无法判断需求值不值得做，也无法做复盘。

### 6.4 这次不做什么

“不做什么”与“做什么”同等重要。

范围外的明确声明，会直接减少争议。

### 6.5 有没有依赖和前置条件

例如：

- 依赖用户中心接口返回用户角色
- 依赖支付系统提供退款状态查询
- 依赖设计稿确认字段展示

把依赖提前写清楚，排期才有意义。

## 7. PRD 的标准结构

下面是一份适用于大多数软件项目的 PRD 结构。

1. 文档信息
2. 背景与问题定义
3. 目标与非目标
4. 用户与角色
5. 使用场景与用户故事
6. 范围定义
7. 功能需求
8. 业务规则
9. 非功能需求
10. 数据与状态变化
11. 异常与边界情况
12. 验收标准
13. 指标与成功标准
14. 依赖、假设、风险
15. 发布与上线要求
16. 待确认问题
17. 附录

你不一定每次都要写得一样长，但逻辑上最好都考虑一遍。

## 8. 每一章到底该写什么

这一章是 `prd.md` 的核心。不是只给目录，而是说明每一节真正该怎么写。

### 8.1 文档信息

建议包含：

- 文档标题
- 版本号
- 作者
- 评审人
- 创建时间
- 最近更新时间
- 当前状态

当前状态可以约定为：

- 草稿
- 待评审
- 评审中
- 已确认
- 已废弃

这一节的作用不是形式主义，而是让团队知道文档是否已经稳定。

### 8.2 背景与问题定义

这一节要回答：`为什么会有这个需求。`

建议写法：

- 当前业务流程是什么
- 当前问题出现在哪里
- 问题影响了谁
- 造成了什么成本或损失
- 为什么现在要处理

错误写法：

- 为了提升用户体验，新增一个筛选功能

这句话太空。更好的写法是：

当前订单后台仅支持按订单编号搜索，不支持按订单状态筛选。运营人员在处理异常订单时，需要人工翻页查找“待审核”订单，平均单次定位耗时约 3 分钟，导致高峰期处理积压明显。因此需要新增按订单状态筛选能力，以提升运营处理效率。

这段话清楚交代了：

- 当前现状
- 谁痛
- 怎么痛
- 为什么要做

### 8.3 目标与非目标

这一节必须同时写“要做什么”和“不做什么”。

目标示例：

- 支持运营人员按订单状态筛选订单
- 将待处理订单定位时间从平均 3 分钟降低到 30 秒以内

非目标示例：

- 本期不调整订单详情页信息结构
- 本期不支持跨店铺联合筛选
- 本期不涉及订单导出功能

非目标很重要，因为它是范围边界的一部分。

### 8.4 用户与角色

建议按角色分别写：

- 角色名称
- 角色说明
- 关键诉求
- 权限边界

示例：

角色：运营人员

- 说明：负责处理订单审核、异常跟进与状态流转
- 关键诉求：快速定位目标订单并执行操作
- 权限边界：可查看所属业务线订单，可修改待审核订单状态

角色：管理员

- 说明：负责系统配置与权限管理
- 关键诉求：控制可见范围与审计
- 权限边界：可配置筛选条件权限，但不直接处理订单

### 8.5 使用场景与用户故事

这里不是写“用户很忙，所以需要高效”，而是写具体场景。

示例场景：

- 每天早上运营登录后台，优先处理前一天遗留的待审核订单
- 客服接到投诉后，需要快速定位退款中的订单状态

基于场景，再写 User Story：

- 作为运营人员，我想按订单状态筛选订单，从而快速找到待审核订单。
- 作为客服人员，我想筛选退款中的订单，从而快速响应用户咨询。

注意，User Story 是价值表达，不是实现细节。

### 8.6 范围定义

这一节建议拆成两部分：

- 本期范围内
- 本期范围外

示例：

本期范围内：

- 订单列表新增状态筛选控件
- 支持按单个订单状态筛选
- 支持状态筛选与订单编号搜索组合使用

本期范围外：

- 不支持多状态同时筛选
- 不支持导出筛选结果
- 不调整订单状态定义本身

写这节时要尽量“硬边界”，不要写模糊表述。

### 8.7 功能需求

这一节建议按编号写。每个功能点都要独立编号，便于评审、拆解、测试和追踪。

建议格式：

- 编号
- 功能名称
- 目标角色
- 前置条件
- 触发方式
- 主流程
- 分支流程
- 后置结果

例如：

编号：FR-1

功能名称：按订单状态筛选订单

目标角色：运营人员、客服人员

前置条件：

- 用户已登录后台
- 用户具有订单列表查看权限

触发方式：

- 用户在订单列表页选择一个订单状态并点击“搜索”

主流程：

1. 系统展示订单状态下拉选项。
2. 用户选择某一状态。
3. 用户点击“搜索”。
4. 系统按所选状态过滤订单列表并返回结果。

分支流程：

1. 若用户未选择状态，则按当前其他筛选条件查询。
2. 若状态值非法，则系统提示“筛选条件无效”。

后置结果：

- 结果列表仅包含符合状态条件的订单。

### 8.8 业务规则

功能告诉团队“做什么”，业务规则告诉团队“必须遵守什么”。

常见业务规则包括：

- 状态流转规则
- 资格规则
- 计算规则
- 限制规则
- 优先级规则

示例：

- BR-1：仅状态为“待审核”的订单允许执行审核通过操作。
- BR-2：已关闭订单不得发起退款申请。
- BR-3：退款金额不得高于用户实际支付金额。

业务规则建议独立编号，因为它经常被多个功能复用。

### 8.9 非功能需求

新人最容易漏掉这一章，但真实项目里它非常重要。

非功能需求主要包括：

- 性能
- 安全
- 可用性
- 稳定性
- 兼容性
- 可观测性
- 合规性

示例：

- NFR-1：订单筛选请求在 95% 情况下应于 2 秒内返回。
- NFR-2：无权限用户访问订单列表接口时应返回无权限错误，不得返回部分数据。
- NFR-3：关键筛选操作应记录审计日志，包括用户 ID、筛选条件、时间戳。

好的非功能需求必须可衡量，不要写“性能要好”“安全要加强”。

### 8.10 数据与状态变化

这一节常被忽略，但对开发和测试都很重要。

建议写清楚：

- 涉及哪些核心实体
- 关键字段有哪些
- 状态如何变化
- 变化由什么事件触发
- 哪些系统会受影响

示例：

核心实体：订单

关键字段：

- order_id
- order_status
- refund_status
- updated_at

状态变化示例：

- 当审核通过时，订单状态从“待审核”变为“处理中”
- 当退款申请提交后，退款状态从“无”变为“申请中”

如果需求涉及状态机，最好画状态流转图；如果暂时不画图，也至少要把状态变化写成列表。

### 8.11 异常与边界情况

这一节决定了你的文档像不像真正能落地的工程文档。

至少应考虑：

- 输入为空
- 输入非法
- 数据不存在
- 权限不足
- 状态不允许
- 外部系统失败
- 重复提交
- 并发冲突
- 超时
- 部分成功、部分失败

示例：

- EC-1：若用户选择的状态值已废弃，系统应提示“筛选条件已失效”，并阻止查询。
- EC-2：若订单列表服务超时超过 5 秒，前端应提示“查询超时，请稍后重试”。
- EC-3：若用户无权查看某店铺订单，则查询结果中不得出现该店铺数据。

很多线上问题，都是因为 PRD 没有把异常场景定义清楚。

### 8.12 验收标准

验收标准是 PRD 是否真正可执行的关键。

写验收标准时，推荐使用两种方式之一：

- 列表式条件
- Given / When / Then

示例：

AC-1：

- Given 运营人员已登录订单后台
- When 其选择“待审核”状态并执行搜索
- Then 系统应只返回状态为“待审核”的订单

AC-2：

- Given 用户无订单查看权限
- When 其访问订单列表页
- Then 系统应提示无权限，且不展示订单数据

验收标准不应该写成：

- 支持筛选
- 体验良好
- 查询正常

因为这些都不可测试。

### 8.13 指标与成功标准

这一节回答“上线后怎么判断做得值不值”。

可以分为两类指标：

- 业务指标
- 系统指标

业务指标示例：

- 运营定位目标订单平均耗时降低 80%
- 人工筛单操作步数从 8 步降到 3 步

系统指标示例：

- 订单筛选接口 P95 响应时间低于 2 秒
- 筛选失败率低于 0.5%

### 8.14 依赖、假设、风险

建议分开写。

依赖示例：

- 依赖订单服务提供标准状态枚举
- 依赖权限系统返回当前用户可访问店铺集合

假设示例：

- 假设现有订单状态枚举在本期内不发生结构性变更

风险示例：

- 若订单状态定义在开发中期调整，筛选逻辑和测试用例需要同步返工

### 8.15 发布与上线要求

不是每个需求都必须写很多，但大型需求最好写。

可包含：

- 是否灰度发布
- 是否加功能开关
- 是否需要数据迁移
- 上线前检查项
- 回滚方案

示例：

- 本功能需通过开关控制，首批仅对运营团队开放
- 若上线后查询失败率超过 2%，应关闭开关并回退到旧筛选逻辑

### 8.16 待确认问题

任何没定的东西都不要藏起来。

建议显式列出：

- 问题内容
- 责任人
- 预期确认时间
- 若未确认，对排期或实现的影响

这可以避免团队误以为需求已经稳定。

### 8.17 附录

可放：

- 术语解释
- 参考链接
- 原型图链接
- 流程图
- 历史讨论结论

附录不是凑页数，而是给团队补充上下文。

## 9. 如何把模糊需求写成可执行需求

这是新人最需要训练的能力。

下面看几个例子。

### 9.1 从“支持退款”到可执行需求

模糊说法：

- 支持退款

这句话至少还要追问：

- 谁能发起退款
- 哪些订单能退款
- 是否全额和部分退款都支持
- 是否需要审核
- 退款失败怎么处理
- 退款状态如何展示
- 与支付系统如何同步

收敛后的需求表达示例：

- 已支付且未完成售后的订单支持发起退款申请。
- 仅客服和运营角色可发起退款申请。
- 单次退款金额不得超过实付金额。
- 退款申请提交后，系统将退款状态更新为“申请中”。
- 若支付系统返回失败，系统应展示失败原因并允许重新提交。

这才是逐渐接近可实现需求的写法。

### 9.2 从“优化搜索”到可执行需求

模糊说法：

- 优化搜索体验

应该继续拆问：

- 搜索慢，还是搜不到，还是结果不准
- 针对哪个页面
- 针对哪类用户
- 优化目标是什么

可能收敛为：

- 在订单列表页新增按订单状态筛选能力
- 订单编号搜索与状态筛选可组合使用
- 搜索结果应在 P95 2 秒内返回

注意，真正可执行的需求不是形容词，而是行为和约束。

## 10. 如何写功能需求

功能需求是 PRD 最常见的主体部分。

### 10.1 功能需求的推荐写法

每个功能点尽量采用统一结构：

- 功能编号
- 功能名称
- 目标
- 角色
- 前置条件
- 触发方式
- 主流程
- 分支流程
- 结果
- 验收关联

统一结构的好处是：

- 团队更容易阅读
- 评审更容易发现缺项
- 测试更容易映射用例

### 10.2 功能需求不要混入太多实现细节

需求写法应尽量聚焦“系统行为”，而不是“代码实现”。

不推荐：

- 前端用防抖 300ms 调接口
- 后端用联合索引优化查询

推荐：

- 用户连续修改筛选条件时，系统不应产生明显卡顿
- 搜索结果应在性能目标范围内返回

当然，如果你的团队流程要求产品也要写接口草案，那可以补充，但要明确这是“补充规格”而不是“技术设计结论”。

## 11. 如何写业务规则

业务规则和功能需求很容易混在一起，但最好分开。

### 11.1 为什么要单独写业务规则

因为业务规则通常被多个功能复用。

例如：

- “已关闭订单不得退款”

它可能影响：

- 退款申请按钮是否显示
- 提交接口是否校验
- 列表是否允许批量操作
- 测试用例设计

如果这个规则只藏在某个页面描述里，团队很容易漏掉。

### 11.2 业务规则的几种常见类型

资格类规则：

- 什么人可以做什么事

状态类规则：

- 在什么状态下允许或禁止某动作

计算类规则：

- 金额、积分、税费如何计算

时效类规则：

- 超过多久不允许操作

优先级类规则：

- 多个规则冲突时谁优先

### 11.3 业务规则的好写法

推荐写成明确判断句：

- BR-1：仅已支付订单允许发起退款申请。
- BR-2：同一订单在退款处理中时，不得再次提交退款申请。
- BR-3：优惠券抵扣金额不计入可退款金额。

这样的写法很适合直接转成代码条件和测试条件。

## 12. 如何写非功能需求

很多新人以为非功能需求是“高级一点的东西”，其实不是。它是正式项目里经常决定成败的内容。

### 12.1 性能

要写可测指标。

不要写：

- 查询要快

要写：

- NFR-1：订单列表查询接口 P95 响应时间应低于 2 秒。

### 12.2 安全

要写访问控制、数据保护、审计要求。

例如：

- NFR-2：用户仅可查看其有权限范围内的订单数据。
- NFR-3：涉及手机号的展示必须做脱敏处理。
- NFR-4：关键操作应记录审计日志。

### 12.3 可用性与稳定性

例如：

- NFR-5：当依赖接口失败时，系统应返回明确错误提示，不得长时间无响应。
- NFR-6：发布后核心查询成功率应不低于 99.5%。

### 12.4 可观测性

这是很多团队后期补锅时才意识到的重要项。

例如：

- NFR-7：筛选失败事件应上报监控系统。
- NFR-8：关键接口应记录 trace_id 以支持问题排查。

## 13. 如何写异常流程和边界条件

如果你想让新人快速提升，最好的训练方式之一，就是逼自己系统性思考异常和边界。

### 13.1 异常流程的常见来源

- 用户输错
- 用户乱点
- 数据缺失
- 权限错误
- 状态变化
- 外部依赖失败
- 系统超时
- 并发操作

### 13.2 边界条件的常见类型

数值边界：

- 最小值
- 最大值
- 超出范围

状态边界：

- 刚好允许
- 刚好禁止

权限边界：

- 有权限
- 无权限
- 部分权限

时间边界：

- 截止前一秒
- 截止后一秒

数据边界：

- 空数据
- 单条数据
- 海量数据

### 13.3 写异常时的一个简单方法

对每个核心动作，强制自己问五个问题：

- 如果输入不合法怎么办
- 如果状态不对怎么办
- 如果权限不够怎么办
- 如果依赖失败怎么办
- 如果用户重复操作怎么办

这五问能覆盖大量漏项。

## 14. 如何写验收标准

验收标准必须做到“测试拿去就能写用例”。

### 14.1 不合格的验收标准

- 支持查询
- 页面正常显示
- 体验更好

这些话都不具备验收价值。

### 14.2 合格的验收标准

示例：

- AC-1：当用户选择有效订单状态并点击搜索时，系统应仅返回符合该状态的订单。
- AC-2：当用户无权限访问订单列表时，系统应返回无权限提示，且不显示任何订单数据。
- AC-3：当订单服务超时超过 5 秒时，系统应提示查询超时，并允许用户重试。

### 14.3 为什么推荐 Given / When / Then

因为这种写法天然区分了：

- 前提
- 动作
- 结果

它非常适合从需求收敛到测试，也和 BDD 的表达方式一致。

## 15. PRD 常见错误

### 15.1 只写 happy path

只写正常流程，不写异常和边界，最后一定在线上补课。

### 15.2 没有编号

没有编号意味着：

- 评审时不好引用
- 测试不好映射
- 变更不好追踪

### 15.3 术语不统一

前面写“订单状态”，后面写“交易状态”，再后面写“流程状态”，如果其实是同一个东西，会非常混乱。

### 15.4 范围和目标混在一起

“提升效率”是目标，不是范围。

“新增状态筛选”是范围，不是目标。

### 15.5 把界面细节当成业务定义

界面只是表现，不是需求本体。

### 15.6 没有写不做什么

不写范围外，项目边界就会不断漂移。

### 15.7 需求和实现耦合过深

过早规定实现细节，会压缩技术方案空间，也容易在方案变化时让 PRD 很快过时。

## 16. 评审 PRD 时怎么评

评审不是“大家看过了”，而是系统性检查需求质量。

可以用下面这份检查清单。

### 16.1 问题是否定义清楚

- 当前问题是否真实存在
- 问题影响是否明确
- 目标是否合理

### 16.2 范围是否清楚

- 本期做什么是否明确
- 本期不做什么是否明确
- 有没有隐藏需求混入

### 16.3 规则是否完整

- 主流程是否清楚
- 异常流程是否覆盖
- 业务规则是否冲突

### 16.4 是否可开发

- 开发能否据此拆分任务
- 关键输入输出是否明确
- 外部依赖是否明确

### 16.5 是否可测试

- 每个关键功能是否有验收标准
- 验收标准是否可验证
- 边界场景是否可设计用例

### 16.6 是否可上线

- 是否考虑性能
- 是否考虑权限
- 是否考虑监控和回滚

## 18. 一个从 0 到 1 的完整小例子

最后给一个简化例子，帮助新人把前面的内容串起来。

### 18.1 需求背景

订单后台当前只能按订单编号查询。运营人员在处理积压订单时，无法快速找到“待审核”订单，只能手动翻页查找，效率低。

### 18.2 目标

- 支持按订单状态筛选订单
- 将目标订单定位时间缩短到 30 秒以内

### 18.3 非目标

- 本期不支持多状态组合筛选
- 本期不支持导出筛选结果

### 18.4 用户故事

- 作为运营人员，我想按订单状态筛选订单，从而快速定位待审核订单。

### 18.5 功能需求

FR-1：订单状态筛选

- 角色：运营人员、客服人员
- 前置条件：用户已登录并具有订单查看权限
- 触发方式：在订单列表页选择状态并点击搜索
- 主流程：
  1. 系统展示状态下拉框
  2. 用户选择一个状态
  3. 用户点击搜索
  4. 系统返回符合条件的订单列表
- 分支流程：
  1. 若未选择状态，则保持原查询逻辑
  2. 若状态值失效，则提示筛选条件无效

### 18.6 业务规则

- BR-1：用户仅可查看其权限范围内的订单
- BR-2：状态筛选仅支持单选

### 18.7 非功能需求

- NFR-1：查询接口 P95 响应时间低于 2 秒
- NFR-2：无权限用户不得看到任何订单数据

### 18.8 异常与边界

- EC-1：订单服务超时超过 5 秒时，提示用户稍后重试
- EC-2：筛选结果为空时，应展示空状态，不得报错

### 18.9 验收标准

- AC-1：当运营人员选择“待审核”并搜索时，只返回待审核订单
- AC-2：当无权限用户访问订单列表页时，不显示订单数据并提示无权限
- AC-3：当查询结果为空时，页面展示空状态提示

## 结语

PRD 写作的核心，不是“写得多”，而是“写得能落地”。

一份真正有效的 PRD，必须把模糊意图逐步收敛成明确行为、明确规则、明确边界、明确验收。

如果你写完一份 PRD 后，还能继续回答下面这些问题，那这份文档通常就比较扎实：

- 为什么做这个需求
- 谁在使用
- 具体做什么
- 这次不做什么
- 正常情况怎么工作
- 异常情况怎么处理
- 做成什么样算完成

后续如果继续写整套 spec 教程，建议按 `01 xxx.md`、`02 xxx.md` 的方式拆开，逐步从：

- 需求思维
- PRD 写法
- User Story 写法
- Acceptance Criteria 写法
- 功能 Spec 写法
- 接口 Spec 写法
- 状态机与业务规则写法
- 测试场景设计

这样由浅入深展开，新人会更容易真正学会，而不是只记住几个术语。
