## 1. 为什么要专门讲 Acceptance Criteria

很多新人学需求和规格时，前面都学得挺顺。

他们能理解：

- 什么是需求
- 什么是规格
- 什么是 `PRD`
- 什么是 `User Story`

但一到项目后半段，经常就会出问题。

最常见的争议是：

- 产品说这不算做完
- 开发说功能已经做了
- 测试说不知道按什么标准判断

这时候，真正缺的通常不是更多讨论，而是一开始就没有把“完成定义”写清楚。

而 `Acceptance Criteria`，也就是验收标准，正是用来解决这个问题的。

所以这一章的重点不是告诉你一个格式，而是让你真正理解：

- 为什么验收标准是规格链路中不可少的一环
- 怎样把“完成”写成可验证的定义

## 2. 什么是 Acceptance Criteria

`Acceptance Criteria`，通常翻译为验收标准或验收条件。

它回答的是一个非常直接的问题：

`做到什么程度，才算这个需求完成了。`

注意，这里讲的不是：

- 为什么做
- 用户是谁
- 功能大概是什么

这些前面已经由需求、`User Story`、规格去承担了。

验收标准更聚焦在：

- 哪些行为必须满足
- 哪些结果必须出现
- 哪些条件必须成立

只有满足这些条件，团队才可以说这项需求交付完成。

### 2.1 最简单的一句话理解

你可以先记成：

`Acceptance Criteria` 是把“完成”写成客观标准。`

### 2.2 它为什么很重要

因为没有客观标准，最后所有判断都会变成主观争论。

## 3. 验收标准到底在解决什么问题

验收标准最核心解决的是三类问题。

### 3.1 解决“做没做完”不清楚的问题

如果文档只写：

- 支持退款
- 支持筛选
- 优化搜索

那最后根本无法客观判断：

- 现在这个交付到底算不算完成

### 3.2 解决“测试按什么验”的问题

测试不能靠感觉去判断功能对不对。

测试需要知道：

- 哪些结果必须成立
- 哪些行为必须验证
- 哪些边界必须覆盖

这些都需要验收标准来支撑。

### 3.3 解决“产品、开发、测试理解不一致”的问题

很多时候不是谁不认真，而是每个人脑子里“完成”的定义不一样。

例如：

- 产品觉得空结果也要有明确提示
- 开发觉得只要接口能返回就行
- 测试不知道空结果是否算异常

验收标准的作用，就是尽量把这些分歧前置消化掉。

## 4. 为什么很多团队到了最后总在争“算不算做完”

因为“完成”这个词，如果不写下来，本身就天然含糊。

### 4.1 团队成员关注点不同

- 产品关注需求是否满足业务目标
- 开发关注功能是否已经实现
- 测试关注结果是否可验证

如果没有统一的验收定义，这三种关注点就会在项目后期碰撞。

### 4.2 需求和规格写了很多，但没有落到“完成定义”

有些文档写了背景、功能、流程、规则，但始终少了最后一步：

- 到底做到什么程度才算过

这就像描述了一场比赛规则，但没有写胜负判定条件。

### 4.3 口头约定很容易失效

很多团队习惯：

- 开会时顺口说一下
- 群里补一句
- 临近上线再确认

这会导致：

- 记忆不一致
- 规则遗失
- 后续无法追溯

### 4.4 最后会变成主观判断

没有验收标准时，项目最后最常听到的话就是：

- 差不多了
- 应该可以了
- 看起来没问题

这些话都不是工程标准。

## 5. Acceptance Criteria 和 PRD、Spec、Test Case 的关系

这一节必须分清楚。

### 5.1 和 PRD 的关系

`PRD` 定义需求整体。

它会讲：

- 背景
- 目标
- 范围
- 用户
- 功能
- 规则

验收标准通常会作为 `PRD` 的一部分出现，但它只负责其中一个特定任务：

- 明确交付完成的判断条件

所以：

- `PRD` 可以包含验收标准
- 但验收标准不等于整个 `PRD`

### 5.2 和 Spec 的关系

规格定义：

- 系统应该如何行为
- 边界和异常是什么

验收标准则从这些规格中提炼出：

- 哪些结果必须被验证通过

可以简单理解为：

- `Spec` 讲“系统如何被定义”
- `Acceptance Criteria` 讲“怎样判断这个定义已经满足”

### 5.3 和 Test Case 的关系

这两个最容易混。

验收标准不是测试用例。

区别在于：

- 验收标准定义“要验证什么”
- 测试用例定义“怎么验证”

例如：

验收标准：

- 当运营人员选择“待审核”状态并搜索时，系统应仅返回待审核订单

测试用例会继续展开：

- 用什么账号
- 进入哪个页面
- 点击哪些步骤
- 输入什么数据

### 5.4 一个简单定位

可以记成：

`Acceptance Criteria` 是需求和测试之间的重要桥梁。

## 6. 一条好的验收标准应该具备什么特征

好的验收标准通常至少有六个特征。

### 6.1 清晰

不能含糊，不能靠猜。

### 6.2 具体

要写出条件、动作和结果，而不是只写抽象目标。

### 6.3 可验证

测试或评审时，能判断“满足”还是“不满足”。

### 6.4 与需求相关

验收标准必须服务于这次需求目标，不能写一些和当前需求无关的检查项。

### 6.5 覆盖关键场景

至少要覆盖：

- 核心主流程
- 关键异常
- 关键边界

### 6.6 没有实现细节噪音

验收标准的重点是结果，不是强行规定实现方式。

## 7. 最常见的写法：Given / When / Then

这是最推荐你掌握的一种写法。

### 7.1 基本结构

- `Given`：给定什么前提
- `When`：当发生什么动作
- `Then`：系统应产生什么结果

有时还会加：

- `And`：补充附加结果或条件

### 7.2 为什么这种写法好用

因为它天然把一条验收标准拆成三层：

- 场景前提
- 用户动作或事件
- 预期结果

这会让表达更清楚，也更方便测试转换成用例。

### 7.3 例子

- Given 用户已登录且具有订单查看权限
- When 用户选择“待审核”状态并点击搜索
- Then 系统应仅返回待审核订单

再比如：

- Given 用户无订单查看权限
- When 用户访问订单列表页
- Then 系统应提示无权限
- And 系统不展示任何订单数据

### 7.4 这套写法最适合什么场景

它非常适合：

- 功能行为验收
- 权限控制验收
- 异常处理验收

## 8. 另一种常见写法：列表式验收条件

不是所有团队都用 `Given / When / Then`。

有些团队会更偏爱列表式写法。

### 8.1 示例

- 用户选择有效订单状态并执行查询时，系统应仅返回符合该状态的订单
- 无权限用户访问订单列表页时，系统应提示无权限，且不展示任何订单数据
- 查询结果为空时，系统应展示空状态提示

### 8.2 列表式的优点

- 更简洁
- 适合文档阅读
- 对简单需求足够直接

### 8.3 列表式的风险

如果写得太短，容易丢前提条件。

所以即使用列表式，也要确保：

- 条件清楚
- 动作清楚
- 结果清楚

## 9. 什么样的内容不算合格的验收标准

这一节特别重要。

很多人以为自己写了验收标准，其实写的只是口号。

### 9.1 只有目标，没有判断条件

例如：

- 支持筛选订单
- 支持退款申请

这不是验收标准，只是功能名。

### 9.2 使用模糊词

例如：

- 页面应友好提示
- 系统应合理处理异常
- 查询应比较快

这里的问题是：

- 什么叫友好
- 什么叫合理
- 什么叫比较快

### 9.3 把实现过程写成验收标准

例如：

- 前端应在点击后调用某接口
- 后端应使用某种缓存

这不是验收标准的重点。

验收关心的是结果，不是具体实现方式。

### 9.4 写成主观感受

例如：

- 用户体验更好
- 页面更清爽
- 操作更方便

这些话可以是目标，但不能直接作为验收标准。

## 10. 如何从需求和规格中提炼出验收标准

很多新人知道验收标准重要，但不知道它从哪来。

答案是：

`验收标准不是凭空写出来的，而是从需求目标、规格规则和关键场景里提炼出来的。`

### 10.1 先找关键能力

先问：

- 这次需求最核心要交付的能力是什么

例如：

- 按状态筛选订单

### 10.2 再找关键场景

再问：

- 哪些核心场景必须成立

例如：

- 有权限用户正常筛选
- 无权限用户访问
- 查询结果为空

### 10.3 再找关键规则

再问：

- 哪些规则如果错了，这个需求就不能算通过

例如：

- 仅有权限用户可查看
- 筛选结果必须正确
- 空结果要有明确反馈

### 10.4 再找关键异常和边界

再问：

- 哪些异常或边界场景必须明确验证

例如：

- 非法状态值
- 查询超时
- 无权限访问

### 10.5 最后写成可判断表达

也就是把它们从“规则说明”改写成“验收条件”。

## 11. 功能类验收标准怎么写

功能类验收标准是最常见的一类。

### 11.1 它关注什么

主要关注：

- 用户执行某动作后
- 系统是否产生正确结果

### 11.2 示例

- Given 运营人员已登录并具有订单查看权限
- When 其选择“待审核”状态并点击搜索
- Then 系统应仅返回待审核订单

再例如：

- Given 用户在可退款订单详情页
- When 用户提交退款申请
- Then 系统应创建退款申请并将退款状态更新为“申请中”

### 11.3 功能类验收的重点

- 动作要清楚
- 结果要可判断
- 不要只写“支持某功能”

## 12. 权限类验收标准怎么写

权限是最容易漏、也最容易出问题的一类。

### 12.1 权限类重点关注什么

- 谁能操作
- 谁不能操作
- 无权限时系统如何处理

### 12.2 示例

- Given 用户无订单查看权限
- When 用户访问订单列表页
- Then 系统应提示无权限
- And 系统不展示任何订单数据

再例如：

- Given 用户仅具有查看权限
- When 用户尝试提交退款审核
- Then 系统应阻止操作并提示无操作权限

### 12.3 权限类常见漏项

- 只写允许，不写禁止
- 只写看得见，不写能不能操作

## 13. 异常和边界类验收标准怎么写

很多人写验收标准时只写主流程，这是不够的。

### 13.1 为什么一定要写异常和边界

因为很多关键质量问题都发生在：

- 数据为空
- 权限不足
- 状态不合法
- 外部依赖失败
- 用户重复点击

### 13.2 示例

- Given 用户输入的订单状态值已失效
- When 用户执行搜索
- Then 系统应提示筛选条件无效
- And 系统不执行查询

- Given 订单服务在 5 秒内未返回结果
- When 用户执行订单查询
- Then 系统应提示查询超时
- And 用户可重新发起查询

- Given 订单已处于退款处理中
- When 用户再次提交退款申请
- Then 系统应阻止重复提交并提示当前状态不允许操作

### 13.3 边界类重点

边界类验收标准通常要特别关注：

- 空值
- 最小值 / 最大值
- 状态边界
- 权限边界
- 时间边界

## 14. 非功能类验收标准怎么写

这类验收标准经常被忽略，但非常重要。

### 14.1 常见非功能维度

- 性能
- 安全
- 稳定性
- 可观测性

### 14.2 性能类示例

- 订单状态筛选请求在 95% 情况下应于 2 秒内返回

### 14.3 安全类示例

- 无权限用户访问订单列表接口时，系统不得返回任何订单数据

### 14.4 稳定性类示例

- 当订单服务不可用时，前端应展示明确错误提示，不得长时间无响应

### 14.5 非功能类常见错误

例如：

- 性能要好
- 安全要加强
- 稳定性要提升

这些都不是可验收表达。

## 15. 一个完整例子：从 User Story 和规格写到验收标准

这一节把前面的内容串起来。

### 15.1 User Story

- 作为运营人员，我想按订单状态筛选订单，从而快速定位待审核订单。

### 15.2 规格

- 订单列表页新增状态筛选控件
- 具有订单查看权限的运营和客服可使用该功能
- 筛选仅支持单选
- 用户点击搜索后，系统仅返回符合状态的订单
- 无权限用户不得查看订单数据
- 结果为空时展示空状态

### 15.3 验收标准

- Given 运营人员已登录并具有订单查看权限
- When 其选择“待审核”状态并点击搜索
- Then 系统应仅返回待审核订单

- Given 用户无订单查看权限
- When 用户访问订单列表页
- Then 系统应提示无权限
- And 系统不展示任何订单数据

- Given 查询结果为空
- When 用户完成搜索
- Then 系统应展示空状态提示

通过这组对比，你应该能清楚看到：

- `User Story` 讲价值
- 规格讲定义
- 验收标准讲完成判断

## 16. 常见错误写法与修正

### 16.1 错误写法一

- 支持按状态筛选订单

问题：

- 这不是验收标准，只是功能描述

修正：

- 当具有订单查看权限的用户选择某订单状态并执行查询时，系统应仅返回符合该状态的订单

### 16.2 错误写法二

- 页面要有友好提示

问题：

- 没有说明什么情况下提示、提示什么、达到什么结果

修正：

- 当查询结果为空时，系统应展示空状态提示

### 16.3 错误写法三

- 查询速度要快

问题：

- 没有指标

修正：

- 订单查询请求在 95% 情况下应于 2 秒内返回

### 16.4 错误写法四

- 无权限用户不能操作

问题：

- 不够完整，没有说系统表现

修正：

- 当无权限用户尝试提交退款审核时，系统应阻止操作并提示无操作权限

## 17. 一个可直接复用的模板

下面给一个实用模板。

```md
## Acceptance Criteria

### AC-1 <主流程名称>
- Given <前置条件>
- When <用户动作 / 事件触发>
- Then <预期结果>

### AC-2 <异常或权限场景>
- Given <前置条件>
- When <用户动作 / 事件触发>
- Then <预期结果>
- And <补充结果>

### AC-3 <非功能要求>
- <带指标的验收要求>
```

如果不用 `Given / When / Then`，也可以用列表式：

```md
## Acceptance Criteria

- 当 <前提 + 动作> 时，系统应 <结果>
- 当 <异常 / 权限 / 边界场景> 时，系统应 <结果>
- <性能 / 安全 / 稳定性指标>
```

关键不在于格式，而在于它是否：

- 清楚
- 具体
- 可验证

## 18. 这一章你应该带走什么

读完这一章，你至少应该记住下面几点。

第一，`Acceptance Criteria` 的核心作用，是把“完成”写成客观可判断的标准。

第二，验收标准不能停留在：

- 支持
- 优化
- 正常
- 友好

这种模糊表达上。

第三，一条好的验收标准，至少要做到：

- 条件清楚
- 动作清楚
- 结果清楚
- 可以验证

第四，最实用的写法是：

- `Given / When / Then`

第五，验收标准不是测试用例，但它会直接决定测试应该验证什么。

## 小结

这一章最关键的判断是：

`需求写到最后，如果没有明确的完成标准，就还没有真正闭环。`

而 `Acceptance Criteria`，就是让需求闭环的那一步。
